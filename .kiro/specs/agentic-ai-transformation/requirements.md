# Requirements Document

## Introduction

This document specifies the requirements for transforming the Runsheet logistics platform from an agent-augmented system (read-only AI assistant) into a fully agentic AI system where AI agents autonomously monitor, decide, and act on operational data. The current platform has a single Strands-based AI agent with 26 read-only tools that answers natural language queries. This transformation gives agents write access with confirmation protocols, adds autonomous monitoring agents that act without human prompting, introduces multi-step goal planning, establishes a multi-agent architecture with specialized domain agents coordinated by an orchestrator, shifts the frontend to an agent-first interface, and adds long-term memory and learning from human feedback.

The transformation builds on the existing infrastructure: Strands SDK agent with Gemini 2.5 Flash, FastAPI backend, Elasticsearch data layer, WebSocket real-time updates, Redis for state management, and the existing tool ecosystem (fleet, orders, inventory, ops intelligence, fuel monitoring, scheduling).

## Glossary

- **Mutation_Tool**: An AI agent tool function that modifies data in Elasticsearch or triggers side effects (job assignment, status change, fuel refill request, rider reassignment). Contrast with existing read-only tools.
- **Confirmation_Protocol**: The mechanism by which the agent presents a proposed action to the user and waits for explicit approval before executing high-impact mutations. Low-impact actions may execute immediately based on configurable risk classification.
- **Action_Risk_Level**: Classification of a mutation as low, medium, or high risk. Low-risk actions (adding a note, updating an ETA) execute immediately. Medium-risk actions (starting a job, recording fuel) require brief confirmation. High-risk actions (cancellation, reassignment, bulk operations) require explicit user approval.
- **Approval_Queue**: A persistent queue of proposed agent actions awaiting human approval, stored in Elasticsearch with status tracking (pending, approved, rejected, expired, executed).
- **Autonomous_Agent**: A background agent process that runs continuously, monitoring data streams and acting on predefined conditions without requiring a user chat message to trigger. Contrast with the existing chat-driven agent.
- **Monitoring_Rule**: A condition-action pair that defines when an autonomous agent should act. Example: "When a job is delayed > 30 minutes, find an alternative asset and propose reassignment."
- **Agent_Orchestrator**: The top-level agent that receives user requests, decomposes them into subtasks, delegates to specialist agents, and synthesizes their results into a coherent response.
- **Specialist_Agent**: A domain-focused agent with a narrow tool set and system prompt optimized for a specific area (fleet, scheduling, fuel, ops intelligence, reporting).
- **Execution_Plan**: A structured multi-step plan generated by the orchestrator or a specialist agent to accomplish a complex goal. Contains ordered steps, dependencies, rollback actions, and success criteria.
- **Agent_Activity_Log**: A persistent, queryable record of all autonomous agent actions, decisions, and outcomes stored in a dedicated Elasticsearch index for observability and audit.
- **Agent_Memory_Store**: A long-term memory system (Elasticsearch-backed) that stores operational patterns, user preferences, and learned behaviors beyond session-scoped conversation history.
- **Feedback_Signal**: A human override or correction of an agent decision that is stored as a learning signal to improve future autonomous decisions.
- **Agent_Autonomy_Level**: A per-tenant configurable setting that controls how much authority agents have: "suggest-only" (current behavior), "auto-low" (auto-execute low-risk, confirm medium/high), "auto-medium" (auto-execute low+medium, confirm high), "full-auto" (auto-execute all with audit logging).

## Requirements

### Requirement 1: Mutation Tools with Risk-Classified Confirmation Protocol

**User Story:** As an operations manager, I want the AI agent to execute actions on my behalf (assign assets, change job status, trigger refills) with appropriate confirmation based on risk level, so that I can operate through natural language while maintaining control over high-impact decisions.

#### Acceptance Criteria

1. THE AI_Agent SHALL expose mutation tools for scheduling operations: `assign_asset_to_job(job_id, asset_id)`, `update_job_status(job_id, new_status, reason)`, `cancel_job(job_id, reason)`, `create_job(job_type, origin, destination, scheduled_time, asset_id, cargo_manifest)`
2. THE AI_Agent SHALL expose mutation tools for ops operations: `reassign_rider(shipment_id, new_rider_id, reason)`, `escalate_shipment(shipment_id, priority, reason)`
3. THE AI_Agent SHALL expose mutation tools for fuel operations: `request_fuel_refill(station_id, quantity_liters, priority)`, `update_fuel_threshold(station_id, threshold_pct)`
4. WHEN a mutation tool is invoked, THE AI_Agent SHALL classify the action risk level as low, medium, or high based on a configurable risk classification registry
5. WHEN a low-risk action is invoked (adding notes, updating ETAs, updating thresholds), THE AI_Agent SHALL execute immediately and report the result to the user
6. WHEN a medium-risk action is invoked (starting a job, recording fuel consumption), THE AI_Agent SHALL present a brief summary and execute after a 5-second confirmation window unless the user cancels
7. WHEN a high-risk action is invoked (cancellation, reassignment, bulk operations), THE AI_Agent SHALL present the full action plan with impact analysis and wait for explicit user approval before executing
8. WHEN any mutation is executed, THE AI_Agent SHALL log the action to the Agent_Activity_Log with actor_id "ai_agent", the tool name, parameters, risk level, confirmation method, and outcome
9. THE AI_Agent SHALL validate all mutation parameters against the same business rules as the existing REST API endpoints (e.g., valid status transitions, asset compatibility, stock limits) before executing
10. IF a mutation fails validation or execution, THEN THE AI_Agent SHALL report the failure reason to the user and suggest corrective actions without retrying automatically

### Requirement 2: Approval Queue for Pending Agent Actions

**User Story:** As an operations manager, I want a queue of pending agent-proposed actions that I can approve or reject, so that high-impact decisions are never executed without my explicit consent.

#### Acceptance Criteria

1. THE Backend_Service SHALL maintain an `agent_approval_queue` Elasticsearch index storing proposed actions with fields: action_id, action_type, tool_name, parameters, risk_level, proposed_by (agent_id), proposed_at, status (pending, approved, rejected, expired, executed), reviewed_by, reviewed_at, expiry_time, impact_summary, and tenant_id
2. WHEN a high-risk action requires approval, THE AI_Agent SHALL create an entry in the Approval_Queue with status "pending" and a configurable expiry time (default 1 hour)
3. THE Backend_Service SHALL expose a GET `/agent/approvals` endpoint that returns pending approval requests for the authenticated tenant, sorted by proposed_at descending
4. THE Backend_Service SHALL expose a POST `/agent/approvals/{action_id}/approve` endpoint that marks the action as approved and triggers execution through the original mutation tool
5. THE Backend_Service SHALL expose a POST `/agent/approvals/{action_id}/reject` endpoint that marks the action as rejected with an optional rejection reason
6. WHEN an approval expires without review, THE Backend_Service SHALL mark it as "expired" and log the expiry to the Agent_Activity_Log
7. THE Backend_Service SHALL broadcast approval queue changes via WebSocket so the frontend updates in real time
8. WHEN an approved action is executed, THE Backend_Service SHALL log the full execution result (success/failure) back to the approval queue entry

### Requirement 3: Autonomous Delay Response Agent

**User Story:** As an operations commander, I want an autonomous agent that monitors for delayed jobs and automatically proposes or executes corrective actions, so that delays are addressed proactively without waiting for a human to notice and intervene.

#### Acceptance Criteria

1. THE Backend_Service SHALL run a Delay_Response_Agent as a background asyncio task during the application lifespan
2. THE Delay_Response_Agent SHALL poll the `jobs_current` index every 60 seconds (configurable) for jobs where status is "in_progress" and current time exceeds estimated_arrival
3. WHEN a newly delayed job is detected, THE Delay_Response_Agent SHALL query available assets compatible with the job type using the existing `find_available_assets` tool
4. IF a compatible available asset is found, THEN THE Delay_Response_Agent SHALL propose a reassignment action to the Approval_Queue (or execute immediately if the tenant's Agent_Autonomy_Level permits)
5. IF no compatible asset is available, THEN THE Delay_Response_Agent SHALL escalate by logging a WARN alert and broadcasting a delay_alert via WebSocket with the job details and "no_alternative_available" reason
6. THE Delay_Response_Agent SHALL not propose duplicate actions for the same delayed job within a configurable cooldown period (default 15 minutes)
7. THE Delay_Response_Agent SHALL log all monitoring cycles, detections, and actions to the Agent_Activity_Log with agent_id "delay_response_agent"
8. THE Delay_Response_Agent SHALL respect tenant feature flags and skip processing for disabled tenants

### Requirement 4: Autonomous Fuel Management Agent

**User Story:** As a fuel operations coordinator, I want an autonomous agent that monitors fuel station levels and automatically triggers refill requests when stations reach critical levels, so that fuel stockouts are prevented without manual monitoring.

#### Acceptance Criteria

1. THE Backend_Service SHALL run a Fuel_Management_Agent as a background asyncio task during the application lifespan
2. THE Fuel_Management_Agent SHALL poll the `fuel_stations` index every 5 minutes (configurable) for stations where status is "critical" or days_until_empty is less than the configured threshold
3. WHEN a station reaches critical level, THE Fuel_Management_Agent SHALL create a fuel refill request using the `request_fuel_refill` mutation tool with a calculated quantity to restore stock to 80% capacity
4. THE Fuel_Management_Agent SHALL not create duplicate refill requests for the same station within a configurable cooldown period (default 2 hours)
5. WHEN a refill request is created, THE Fuel_Management_Agent SHALL broadcast a fuel_alert via WebSocket with the station details, requested quantity, and urgency level
6. THE Fuel_Management_Agent SHALL log all monitoring cycles, detections, and refill requests to the Agent_Activity_Log with agent_id "fuel_management_agent"
7. THE Fuel_Management_Agent SHALL calculate refill priority based on days_until_empty: critical (< 1 day), high (< 3 days), medium (< 5 days)

### Requirement 5: Autonomous SLA Guardian Agent

**User Story:** As an operations manager, I want an autonomous agent that monitors shipments approaching SLA breach and takes proactive corrective action, so that SLA violations are minimized through early intervention.

#### Acceptance Criteria

1. THE Backend_Service SHALL run an SLA_Guardian_Agent as a background asyncio task during the application lifespan
2. THE SLA_Guardian_Agent SHALL poll the `shipments_current` index every 2 minutes (configurable) for shipments where status is "in_transit" and estimated_delivery is within 30 minutes of the current time (configurable threshold)
3. WHEN a shipment is approaching SLA breach, THE SLA_Guardian_Agent SHALL evaluate the assigned rider's current workload and location
4. IF the assigned rider has more than 3 active shipments (configurable), THEN THE SLA_Guardian_Agent SHALL propose a rider reassignment to a less loaded rider via the Approval_Queue
5. THE SLA_Guardian_Agent SHALL escalate shipments that breach SLA by updating the shipment priority to "critical" and broadcasting an sla_breach event via WebSocket
6. THE SLA_Guardian_Agent SHALL log all monitoring cycles, near-breach detections, and corrective actions to the Agent_Activity_Log with agent_id "sla_guardian_agent"
7. THE SLA_Guardian_Agent SHALL not propose duplicate corrective actions for the same shipment within a configurable cooldown period (default 10 minutes)

### Requirement 6: Multi-Step Execution Planning

**User Story:** As an operations manager, I want the AI agent to decompose complex requests into multi-step plans with dependency tracking and error recovery, so that I can issue high-level commands and the agent handles the orchestration.

#### Acceptance Criteria

1. WHEN the AI_Agent receives a complex request that requires multiple tool invocations, THE Agent_Orchestrator SHALL generate an Execution_Plan with ordered steps, dependencies between steps, and expected outcomes
2. THE Execution_Plan SHALL be presented to the user before execution begins, showing each step, its purpose, and the overall goal
3. WHEN executing a plan, THE Agent_Orchestrator SHALL execute steps in dependency order, passing outputs from completed steps as inputs to dependent steps
4. IF a step in the plan fails, THE Agent_Orchestrator SHALL evaluate whether the failure is recoverable (retry with different parameters) or terminal (abort remaining steps)
5. WHEN a step fails and recovery is possible, THE Agent_Orchestrator SHALL attempt an alternative approach (e.g., if the preferred asset is unavailable, try the next best option) up to 2 recovery attempts per step
6. WHEN a plan completes (successfully or with partial failures), THE Agent_Orchestrator SHALL present a summary showing completed steps, skipped steps, failures, and overall outcome
7. THE Agent_Orchestrator SHALL log the full plan, execution trace, and outcome to the Agent_Activity_Log
8. THE Execution_Plan SHALL support rollback actions for steps that have already been executed if a later step fails and the user requests rollback

### Requirement 7: Multi-Agent Architecture with Orchestrator

**User Story:** As a platform architect, I want specialized domain agents coordinated by an orchestrator, so that each agent has focused expertise and the system scales without degrading accuracy as tool count grows.

#### Acceptance Criteria

1. THE Backend_Service SHALL implement a Fleet_Agent with tools limited to: search_fleet_data, get_fleet_summary, find_truck_by_id, get_all_locations, and fleet mutation tools
2. THE Backend_Service SHALL implement a Scheduling_Agent with tools limited to: search_jobs, get_job_details, find_available_assets, get_scheduling_summary, generate_dispatch_report, and scheduling mutation tools
3. THE Backend_Service SHALL implement a Fuel_Agent with tools limited to: search_fuel_stations, get_fuel_summary, get_fuel_consumption_history, generate_fuel_report, and fuel mutation tools
4. THE Backend_Service SHALL implement an Ops_Intelligence_Agent with tools limited to: search_shipments, search_riders, get_shipment_events, get_ops_metrics, and ops report/mutation tools
5. THE Backend_Service SHALL implement a Reporting_Agent with tools limited to: all report generation tools across domains and cross-domain analytics queries
6. THE Agent_Orchestrator SHALL receive user requests, determine which specialist agent(s) are needed, delegate subtasks, and synthesize results into a unified response
7. WHEN a request spans multiple domains (e.g., "reassign delayed cargo jobs and check fuel levels for the replacement trucks"), THE Agent_Orchestrator SHALL coordinate multiple specialist agents and merge their outputs
8. THE Agent_Orchestrator SHALL maintain a routing table that maps request intents to specialist agents based on keyword matching and intent classification
9. EACH Specialist_Agent SHALL have its own Strands Agent instance with a domain-specific system prompt and tool set, sharing the same Gemini model configuration

### Requirement 8: Agent Activity Log and Observability

**User Story:** As an SRE, I want a comprehensive activity log of all agent actions, decisions, and autonomous operations, so that I can audit agent behavior, debug issues, and monitor autonomous agent health.

#### Acceptance Criteria

1. THE Backend_Service SHALL maintain an `agent_activity_log` Elasticsearch index with strict mapping including fields: log_id, agent_id, action_type (query, mutation, plan, monitoring_cycle, detection, approval_request), tool_name, parameters, risk_level, outcome (success, failure, pending_approval, rejected), duration_ms, tenant_id, user_id, session_id, timestamp, and details (nested object)
2. WHEN any agent tool is invoked (read or write), THE Agent SHALL log the invocation to the Agent_Activity_Log before returning the result
3. WHEN an autonomous agent completes a monitoring cycle, THE Agent SHALL log the cycle with detection count, actions taken, and cycle duration
4. THE Backend_Service SHALL expose a GET `/agent/activity` endpoint that returns paginated activity log entries with filtering by agent_id, action_type, tenant_id, time range, and outcome
5. THE Backend_Service SHALL expose a GET `/agent/activity/stats` endpoint that returns aggregated statistics: actions per agent, success/failure rates, average response times, and approval rates
6. THE Agent_Activity_Log SHALL apply an ILM policy: warm tier after 30 days, cold tier after 90 days, delete after 365 days
7. THE Backend_Service SHALL broadcast agent activity events via WebSocket on a dedicated `/ws/agent-activity` channel so the frontend can display a real-time agent activity feed

### Requirement 9: Agent-First Frontend Interface

**User Story:** As an operations manager, I want the primary interface to be the AI agent with dashboards serving as monitoring views, so that I can operate the logistics platform through natural language commands while observing agent actions in real time.

#### Acceptance Criteria

1. THE Frontend_Application SHALL display a full-width command interface as the primary view at route `/ops/command`, featuring the AI chat prominently with a collapsible dashboard sidebar
2. THE Frontend_Application SHALL display an agent activity feed panel showing real-time autonomous agent actions (delay responses, fuel refills, SLA interventions) with timestamps and outcomes
3. THE Frontend_Application SHALL display an approval queue panel showing pending actions requiring human approval, with approve/reject buttons and impact summaries
4. THE Frontend_Application SHALL support inline action confirmation within the chat interface: when the agent proposes a medium-risk action, the user can approve or reject directly in the chat flow
5. THE Frontend_Application SHALL display agent health indicators showing the status of each autonomous agent (running, paused, error) and their last activity timestamp
6. THE Frontend_Application SHALL allow users to pause and resume individual autonomous agents via the agent health panel
7. WHEN an autonomous agent takes an action, THE Frontend_Application SHALL display a toast notification with the agent name, action summary, and outcome

### Requirement 10: Agent Autonomy Level Configuration

**User Story:** As a tenant administrator, I want to configure how much authority AI agents have in my tenant, so that I can gradually increase automation as trust is established.

#### Acceptance Criteria

1. THE Backend_Service SHALL support four Agent_Autonomy_Levels per tenant: "suggest-only" (all actions require approval), "auto-low" (low-risk auto-execute, medium/high require approval), "auto-medium" (low+medium auto-execute, high requires approval), "full-auto" (all actions auto-execute with audit logging)
2. THE Backend_Service SHALL store the Agent_Autonomy_Level in the tenant configuration (Redis-backed, same pattern as feature flags)
3. WHEN a mutation tool is invoked, THE Confirmation_Protocol SHALL check the tenant's autonomy level and the action's risk level to determine whether to execute immediately, present for confirmation, or queue for approval
4. THE Backend_Service SHALL expose a PATCH `/agent/config/autonomy` endpoint that updates the tenant's autonomy level, requiring admin-level JWT claims
5. WHEN the autonomy level is changed, THE Backend_Service SHALL log the change to the Agent_Activity_Log with the old level, new level, and the user who made the change
6. THE default Agent_Autonomy_Level for new tenants SHALL be "suggest-only" to ensure safe onboarding

### Requirement 11: Long-Term Agent Memory

**User Story:** As an operations manager, I want the AI agent to remember operational patterns and my preferences across sessions, so that it becomes more effective over time and doesn't repeat the same questions.

#### Acceptance Criteria

1. THE Backend_Service SHALL maintain an `agent_memory` Elasticsearch index storing long-term observations with fields: memory_id, memory_type (pattern, preference, feedback), agent_id, tenant_id, content, confidence_score, created_at, last_accessed, access_count, and tags
2. THE AI_Agent SHALL store operational patterns it discovers during analysis (e.g., "Truck T-1042 is frequently delayed on Route 7", "Station S-12 consumption spikes on Thursdays") as pattern memories with a confidence score
3. THE AI_Agent SHALL store user preferences expressed during conversations (e.g., "always notify me before reassignments", "prefer trucks over vans for cargo jobs") as preference memories
4. WHEN the AI_Agent processes a request, THE Agent SHALL query relevant memories from the Agent_Memory_Store and include them in the context for decision-making
5. THE Backend_Service SHALL expose a GET `/agent/memory` endpoint that returns stored memories for the tenant with filtering by memory_type and tags
6. THE Backend_Service SHALL expose a DELETE `/agent/memory/{memory_id}` endpoint that allows users to remove incorrect or unwanted memories
7. THE Agent_Memory_Store SHALL apply a relevance decay: memories not accessed for 90 days have their confidence_score reduced by 50%, and memories with confidence below 0.1 are automatically purged

### Requirement 12: Human Feedback Learning Loop

**User Story:** As an operations manager, I want the AI agent to learn from my corrections and overrides, so that its autonomous decisions improve over time and align with my operational preferences.

#### Acceptance Criteria

1. WHEN a user rejects an agent-proposed action in the Approval_Queue, THE Backend_Service SHALL store the rejection as a Feedback_Signal with the original proposal, rejection reason, and the action the user took instead (if any)
2. WHEN a user manually performs an action that contradicts a recent agent suggestion, THE Backend_Service SHALL detect the override and store it as a Feedback_Signal
3. THE Backend_Service SHALL maintain a `agent_feedback` Elasticsearch index storing feedback signals with fields: feedback_id, agent_id, action_type, original_proposal, user_action, feedback_type (rejection, override, correction), tenant_id, user_id, timestamp, and context
4. WHEN an autonomous agent makes a decision, THE Agent SHALL query recent feedback signals for similar situations and adjust its behavior accordingly (e.g., if reassignments to a specific asset were repeatedly rejected, deprioritize that asset)
5. THE Backend_Service SHALL expose a GET `/agent/feedback` endpoint that returns feedback signals for the tenant with filtering by agent_id, action_type, and time range
6. THE Backend_Service SHALL expose a GET `/agent/feedback/stats` endpoint that returns feedback statistics: rejection rate per agent, most common rejection reasons, and improvement trends over time
7. THE AI_Agent SHALL include a confidence indicator in its proposals based on historical feedback: actions similar to previously rejected proposals receive a lower confidence score and are more likely to require approval regardless of autonomy level
